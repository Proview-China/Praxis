## 新增需求

### 需求:运行时必须支持前向兼容的 hook 生命周期
运行时必须支持在目标工具执行前与执行后调度 hook，并把 hook 当作 `hooks` 类型工具来审计。hook 生命周期至少必须覆盖 `before_tool` 与 `after_tool` 两个阶段，并允许后续扩展更多事件类型；禁止把 hook 仅作为不可追踪的内部副作用处理。

#### 场景:before hook 成功放行
- **当** 工具执行前配置了 `before_tool` hook 且 hook 返回继续
- **那么** 目标工具必须继续执行
- **那么** 执行记录或证据中必须包含该 hook 的名称、阶段与结果

#### 场景:before hook 拒绝主执行流
- **当** `before_tool` hook 返回阻断决策
- **那么** 目标工具执行必须被中止
- **那么** 主执行结果必须返回 `blocked`

### 需求:after hook 必须可审计且不能吞掉主执行结果
运行时必须在主工具执行结束后尝试运行 `after_tool` hook，并且必须把 hook 结果写入审计证据。`after_tool` hook 禁止覆盖主工具的原始成功/失败状态；如果 hook 自身失败，运行时必须记录 hook 失败信息并保留主执行结果。

#### 场景:主工具成功但 after hook 失败
- **当** 主工具执行成功且 `after_tool` hook 运行失败
- **那么** 主执行结果必须仍保持成功
- **那么** 证据中必须记录 after hook 的失败信息与错误原因

### 需求:hook 执行必须避免默认递归
运行时必须在默认配置下避免 hook 对 hook 的无限递归触发。若当前正在执行的工具本身就是 `hooks` 类型，运行时必须默认跳过同阶段 hook 链，除非显式启用递归。

#### 场景:执行 hooks 类型工具
- **当** 当前执行的目标工具本身属于 `hooks`
- **那么** 运行时必须默认跳过自动注入的 before/after hook 链
- **那么** 审计证据必须说明该次跳过是递归保护行为
