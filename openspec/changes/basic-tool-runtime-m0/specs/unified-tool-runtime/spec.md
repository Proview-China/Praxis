## 新增需求

### 需求:统一工具执行记录必须保留真实工具类别
运行时必须根据注册工具的元数据或已归一化的运行时事件，为每次工具执行记录真实的 `tool_kind`，并且必须支持 `function`、`web`、`code`、`computer`、`shell`、`hooks`、`skills`、`mcp` 这些类别值。运行时禁止把非 `function` 工具一律写成 `function`，否则上层编排器无法正确审计与分流。

#### 场景:注册的 shell 工具被执行
- **当** 已注册工具被声明为 `shell` 类型并成功执行
- **那么** 返回的执行记录必须把 `tool_kind` 记为 `shell`
- **那么** 记录中的 `evidence` 必须至少包含执行器来源与时间戳

#### 场景:provider wrapper 读取已有执行记录
- **当** 已存储的执行记录被再次包装为 provider 输出
- **那么** 执行记录中的 `tool_kind` 必须保持原值不变
- **那么** 运行时禁止在包装阶段把原始类别丢失或重写成 `function`

### 需求:统一运行时必须为多类工具提供最小可复现执行路径
统一运行时必须允许已注册工具通过同一个执行入口被调度，并根据 `tool_kind` 与执行器配置返回结构化结果。对于 `web`、`code`、`computer`、`shell`、`skills`、`mcp` 类型，M0 实现可以使用 builtin 或 fixture adapter，但必须返回可验证证据、结构化状态以及明确错误码；禁止 silent failure。

#### 场景:fixture 模式的 web 工具被调用
- **当** `web` 工具以 fixture adapter 模式执行查询
- **那么** 返回结果必须包含可引用证据集
- **那么** 每条证据必须能够标识来源、标题或 URL 之类的追踪信息

#### 场景:未配置 live adapter 的 computer 工具被调用
- **当** `computer` 工具在没有 live adapter 且没有 fixture adapter 的情况下被执行
- **那么** 运行时必须返回 `blocked` 或 `failed`
- **那么** 错误对象必须包含可诊断的错误码与原因说明

### 需求:POSIX 沙盒执行必须只在 macOS 与 Linux 可用
`shell` 与 `code` builtin 执行器必须只在 macOS 与 Linux 下提供可执行沙盒路径，并且必须在执行记录中保留 `stdout`、`stderr`、`exit_code`、工作目录、执行器证据等可复盘信息。对于非支持平台，运行时必须返回结构化不支持结果，而不是尝试降级为无沙盒执行。

#### 场景:POSIX shell 工具在受限目录内运行
- **当** `shell` 工具在允许的工作目录和允许的命令策略下执行
- **那么** 返回结果必须包含 `stdout`、`stderr` 与 `exit_code`
- **那么** 执行记录必须包含工作目录与执行器类型等审计证据

#### 场景:非支持平台请求 POSIX shell 执行
- **当** 当前平台不是 macOS 或 Linux
- **那么** 运行时必须返回 `blocked`
- **那么** 错误对象必须明确说明该沙盒执行器当前不支持该平台

### 需求:策略校验必须对高风险执行做显式拦截
对于 `shell`、`code`、`computer` 等高风险工具，运行时必须在执行前校验权限策略，并且必须支持最小集合的目录、命令、超时或 adapter 开关校验。策略不满足时，运行时必须返回可审计的 `blocked` 结果，而不是尝试执行后再失败。

#### 场景:命令未被 shell 策略允许
- **当** `shell` 工具请求执行的命令不在允许清单内
- **那么** 运行时必须在执行前返回 `blocked`
- **那么** 错误对象必须标识策略拒绝，而不是底层进程启动失败
