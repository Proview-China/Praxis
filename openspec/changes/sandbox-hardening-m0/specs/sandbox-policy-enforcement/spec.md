## 新增需求

### 需求:沙盒执行必须由 core 统一应用 policy
`shell` 与 `code` 执行器必须由 core 在执行前统一应用 sandbox policy，至少覆盖工作目录、允许命令、允许运行时、网络开关、输出上限与产物数量限制。UI 可以传入策略配置，但禁止在 UI 层自行做安全决策后再调用一个宽松执行器。

#### 场景:允许策略下的 shell 执行
- **当** UI 提交合法命令、合法工作目录并且 policy 允许执行
- **那么** core 必须执行该 shell 请求
- **那么** 返回结果中必须包含实际生效的 policy snapshot

#### 场景:运行时不在允许清单中
- **当** `code` 执行请求使用了不在允许清单中的 runtime
- **那么** core 必须在执行前返回 `blocked`
- **那么** 错误对象必须明确说明被 policy 拒绝

### 需求:网络默认必须显式关闭或显式说明未启用
对于 `shell` 与 `code` 沙盒，network access 必须由 core 统一控制。若当前平台或实现阶段无法真正打开细粒度网络放行，M0 默认必须拒绝带网络需求的请求，或者明确返回“该能力未启用”，禁止静默放行。

#### 场景:请求需要网络但 policy 禁止
- **当** 请求声明需要网络而 sandbox policy 关闭了 network access
- **那么** core 必须返回 `blocked`
- **那么** 错误对象必须说明网络被禁止，而不是让子进程自行失败

### 需求:高风险副作用必须受目录与产物治理限制
沙盒执行必须限制工作目录与产物输出范围。对于不在允许目录中的路径访问或超出产物上限的结果，core 必须在执行前阻断或在返回时裁剪，并把裁剪/阻断行为写入审计证据。

#### 场景:cwd 超出允许范围
- **当** 请求提供的工作目录不在允许目录列表内
- **那么** core 必须返回 `blocked`
- **那么** 审计证据必须说明被拒绝的目录
