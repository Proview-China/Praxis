## 新增需求

### 需求:沙盒执行必须支持超时与中断
`shell` 与 `code` 执行器必须支持由 core 控制的超时和中断语义。超时与中断必须使用独立状态表示，禁止统一折叠成普通失败，以便 UI 和上层编排器正确理解执行结果。

#### 场景:执行超时
- **当** shell 或 code 执行超过 policy 指定的超时
- **那么** core 必须终止该执行
- **那么** 返回状态必须为 `timeout`

#### 场景:执行被中断
- **当** 上层或 UI 请求中断一个正在运行的沙盒执行
- **那么** core 必须尽快终止该执行
- **那么** 返回状态必须为 `interrupted`

### 需求:输出与产物必须受配额限制
沙盒执行结果必须支持 `stdout/stderr` 大小限制与产物数量限制。超出限制时，core 必须裁剪并在结果中记录裁剪元数据，而不是直接把无限制输出交给 UI。

#### 场景:stdout 超出限制
- **当** 执行产生的 stdout 超过配置上限
- **那么** core 必须裁剪 stdout
- **那么** 结果中必须说明 stdout 已被裁剪

#### 场景:产物数量超出上限
- **当** 执行产生的产物数量超过允许上限
- **那么** core 必须按规则裁剪产物清单
- **那么** 审计结果中必须说明有多少产物被省略

### 需求:无法启用的资源限制必须显式告知
对于 CPU 与内存限制等在当前平台或本轮实现中无法可靠启用的控制项，core 必须在结果或审计中显式说明“未启用/未支持”，而不是让调用方误以为限制已生效。

#### 场景:请求配置了 CPU 限制但当前实现不支持
- **当** policy 请求 CPU 限制而当前平台/实现尚未启用该能力
- **那么** 结果中必须明确标注该限制未生效
- **那么** 调用方不得把该执行视为已经受 CPU 限制保护
